# 方案设计文档

## 1. 项目目标

本项目首版目标是跑通一条最核心的 AI 学习闭环：

用户输入学习内容 -> AI 生成题目 -> 用户闯关答题 -> AI 生成学习分析报告。

首版不优先做登录注册、复杂用户体系、多人 PK、排行榜、企业后台和大型知识库，先验证用户是否愿意输入内容、连续答题、查看复盘报告。

## 2. MVP 核心流程

### 2.1 用户输入内容

用户在小程序首页输入想学习的内容，首版优先支持：

- 一句话主题
- 一段文本
- 网页链接
- 文件上传

首版实现优先级：

1. 文本输入
2. 网页链接
3. 文件上传
4. 视频链接

视频解析成本较高，首版只保留入口或延后实现。

### 2.2 AI 生成题目

后端接收用户输入后，调用大模型生成 5 到 10 道题。

题目首版支持：

- 单选题
- 判断题

每道题包含：

- 题干
- 选项
- 正确答案
- 答案解析
- 知识点标签
- 难度

### 2.3 用户答题

用户逐题答题，每次提交答案后立即返回：

- 是否正确
- 正确答案
- 答案解析
- 错误原因
- 当前进度

### 2.4 AI 生成报告

用户完成全部题目后，后端根据题目、答案和错误记录生成复盘报告。

报告包含：

- 本次学习摘要
- 正确率
- 掌握情况
- 错题汇总
- 薄弱知识点
- 下一步学习建议

### 2.5 分享

首版支持分享学习结果页，分享内容包含：

- 学习主题
- 得分
- 正确率
- 简短学习总结

## 3. 技术选型

### 3.1 小程序端技术选型对比

| 方案 | 优点 | 缺点 | 结论 |
|---|---|---|---|
| 微信原生小程序 | 官方能力支持最好，调试稳定，微信生态兼容性强 | 跨端能力弱，工程复用能力有限 | 不采用 |
| Taro | React 技术栈，多端能力强，可编译到微信小程序，后续扩展 H5/多小程序成本更低 | 需要注意小程序环境下的兼容限制 | 首版采用 |
| uni-app | Vue 技术栈，多端生态成熟，上手快 | 平台差异仍需适配，首版会增加工程复杂度 | 暂不推荐首版使用 |

首版采用 Taro 框架开发微信小程序。

原因：

- 使用 React 技术栈提升组件化开发效率。
- 保留后续扩展 H5、支付宝小程序、抖音小程序的空间。
- Taro 提供小程序编译能力和运行时适配能力。
- 首版仍以微信小程序为主要发布目标。

### 3.2 后端技术选型

后端采用：

- Java 17+
- Spring Boot 3
- Spring MVC
- MyBatis-Plus
- MySQL 8
- Spring AI Alibaba
- 通义千问
- Maven

推荐原因：

- Spring Boot 适合快速构建稳定 REST API。
- MyBatis-Plus 适合快速完成 CRUD 和分页查询。
- MySQL 适合保存学习会话、题目、答题记录和报告。
- Spring AI Alibaba 适合在 Spring Boot 项目中封装大模型调用。
- Java 后端生态成熟，后续接入支付、对象存储、消息队列更方便。

### 3.3 AI 调用方案

小程序不直接调用大模型，后端通过 Spring AI Alibaba 调用通义千问。

调用链路：

Taro 微信小程序 -> Spring Boot 后端 -> Spring AI Alibaba -> 通义千问 -> Spring Boot 后端 -> Taro 微信小程序

原因：

- 避免在小程序端暴露 API Key。
- 方便统一限流、日志、重试和异常处理。
- 方便保存生成结果和答题过程。
- 方便后续切换不同模型服务。
- Spring AI Alibaba 可以统一封装 ChatClient、结构化输出和模型参数。

## 4. 系统架构

### 4.1 整体架构

```text
Taro 微信小程序
  |
  | HTTPS
  v
Spring Boot API 服务
  |
  | SQL
  v
MySQL
  |
  | HTTPS
  v
Spring AI Alibaba
  |
  | HTTPS
  v
通义千问
```

### 4.2 后端模块

后端按功能拆分：

- LearningSession：学习会话
- SourceMaterial：输入材料
- Question：题目
- AnswerRecord：答题记录
- LearningReport：学习报告
- AIService：Spring AI Alibaba 调用封装
- ShareService：分享结果

### 4.3 AI 服务模块

AI 服务封装：

- QuestionGenerationService：生成题目
- AnswerEvaluationService：判断答案和生成讲解
- ReportGenerationService：生成复盘报告

底层统一通过 Spring AI Alibaba 调用通义千问。

## 6. 后端接口设计

### 6.1 创建学习会话

```http
POST /api/learning/sessions
```

请求字段：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| inputType | string | 是 | text/url/file/video |
| content | string | 否 | 文本或链接 |
| fileId | long | 否 | 文件 ID |

返回字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| sessionId | long | 学习会话 ID |
| status | string | 会话状态 |

### 6.2 生成题目

```http
POST /api/learning/sessions/{sessionId}/questions
```

返回字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| sessionId | long | 学习会话 ID |
| questions | array | 题目列表 |

### 6.3 提交答案

```http
POST /api/learning/sessions/{sessionId}/answers
```

请求字段：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| questionId | long | 是 | 题目 ID |
| userAnswer | string | 是 | 用户答案 |

返回字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| correct | boolean | 是否正确 |
| correctAnswer | string | 正确答案 |
| explanation | string | 答案解析 |

### 6.4 生成报告

```http
POST /api/learning/sessions/{sessionId}/report
```

返回字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| reportId | long | 报告 ID |
| summary | string | 学习摘要 |
| score | int | 得分 |
| accuracy | decimal | 正确率 |
| weakPoints | array | 薄弱知识点 |
| suggestions | array | 学习建议 |

### 6.5 查询学习会话

```http
GET /api/learning/sessions/{sessionId}
```

返回：

- 学习会话
- 输入材料
- 题目列表
- 答题记录
- 报告

## 8. AI 生成方案

后端通过 Spring AI Alibaba 接入通义千问，统一封装模型调用、提示词、结构化输出、异常重试和日志记录。

推荐配置：

- 使用 DashScope 通义千问模型。
- API Key 只保存在后端环境变量或配置中心。
- 题目生成和报告生成分别配置独立提示词。
- 使用结构化输出对象承接 AI 返回结果。
- AI 原始返回内容和解析后的业务对象分别记录，方便排查问题。

### 8.1 题目生成要求

AI 返回必须是结构化 JSON。

题目 JSON 包含：

- type
- stem
- options
- correctAnswer
- explanation
- knowledgePoint
- difficulty

后端必须校验：

- 是否是合法 JSON
- 题目数量是否符合要求
- 每道题是否有正确答案
- 单选题是否至少有 4 个选项
- 正确答案是否存在于选项中
- 解析是否为空

校验失败时重试一次，仍失败则返回生成失败。

### 8.2 答案判断

首版单选题和判断题不需要每次调用 AI 判断，后端直接根据 `correct_answer` 判断。

AI 主要负责：

- 生成题目
- 生成答案解析
- 生成最终报告

### 8.3 报告生成要求

报告生成输入：

- 原始学习内容
- 题目列表
- 用户答案
- 正确率
- 错题列表

报告输出：

- 学习摘要
- 掌握情况
- 薄弱知识点
- 错题复盘
- 下一步建议

## 10. 风险与应对

### 10.1 题目质量不稳定

应对：

- 固定结构化提示词
- 限制题型
- 后端校验 JSON
- 不合格结果自动重试

### 10.2 AI 成本过高

应对：

- 首版限制输入长度
- 限制题目数量
- 报告只在完成答题后生成
- 缓存生成结果

### 10.3 长文档解析困难

应对：

- 首版限制文件大小
- 优先支持 txt、docx、pdf
- 长内容先摘要再出题
- 视频解析延后

### 10.4 小程序审核风险

应对：

- 避免生成敏感内容
- 后端增加内容安全检查
- 对 AI 输出做敏感词过滤
- 保留用户反馈入口

### 10.5 无登录导致数据归属弱

应对：

- 首版用 sessionId 完成核心流程
- 本地缓存最近学习记录
- 后续再接入微信登录绑定历史数据

## 11. 验收标准

首版完成标准：

- 用户不登录也能完成一次完整学习流程。
- 用户输入一段文本后能生成 5 到 10 道题。
- 用户每答一题都能看到对错和解析。
- 用户完成全部题目后能看到复盘报告。
- MySQL 能完整保存学习会话、题目、答题记录和报告。
- AI 返回异常时有失败提示和重试机制。
- 小程序可以分享学习结果页。
